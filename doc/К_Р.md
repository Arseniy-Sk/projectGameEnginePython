## Глава 1. Теоретические основы разработки игровых фреймворков

### 1.1 Понятие и архитектура игрового фреймворка

**Игровой фреймворк** (game framework) — это программная платформа, предоставляющая разработчику набор инструментов и библиотек для создания видеоигр. В отличие от игрового движка, фреймворк оставляет больше контроля разработчику, предоставляя каркас приложения, который необходимо наполнить конкретной логикой.

Как отмечает Стив Макконнелл в книге «Совершенный код», «хорошая архитектура программного обеспечения должна быть модульной, легко расширяемой и понятной» [1, с. 87]. Эти принципы легли в основу проектирования GameLib.

Классическая архитектура игрового фреймворка включает следующие компоненты:

```
┌─────────────────────────────────────────────────────────────────┐
│                         Игровой цикл                            │
├─────────────────────────────────────────────────────────────────┤
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ Ввод    │ │Обновление│ │Физика   │ │Рендеринг│ │Аудио    │   │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘   │
├─────────────────────────────────────────────────────────────────┤
│                      Система управления сценами                  │
├─────────────────────────────────────────────────────────────────┤
│                    Компонентная система объектов                 │
├─────────────────────────────────────────────────────────────────┤
│                      Библиотеки и зависимости                    │
└─────────────────────────────────────────────────────────────────┘
```

**Рисунок 1.1 — Типовая архитектура игрового фреймворка**

Ключевым элементом любого игрового фреймворка является **игровой цикл**. Это основной цикл программы, который выполняется непрерывно на протяжении всей игры. В работе [2] выделяют три основных типа игровых циклов:

1. **Цикл с фиксированным FPS** — обновление состояния происходит с постоянной частотой.
2. **Цикл с переменным FPS** — обновление привязано к реальному времени.
3. **Гибридный цикл** — комбинация двух подходов.

В GameLib реализован гибридный подход с использованием отдельного потока для игрового цикла и возможностью установки целевого FPS.

### 1.2 Объектно-ориентированные паттерны в геймдеве

Паттерны проектирования (design patterns) — это повторно используемые решения типовых проблем проектирования. Эрик Гамма и соавторы в книге «Приемы объектно-ориентированного проектирования. Паттерны проектирования» [3] выделяют 23 классических паттерна.

В контексте игровой разработки наиболее часто применяются следующие паттерны:

| Паттерн | Применение в играх |
|---------|---------------------|
| **Компоновщик** | Древовидная структура игровых объектов, система компонентов |
| **Наблюдатель** | Система событий, достижения, звуковые уведомления |
| **Стратегия** | Поведение ИИ, различные типы физики, камера |
| **Фабричный метод** | Создание врагов, пулы объектов, загрузка уровней |
| **Одиночка** | Глобальные менеджеры: звук, ресурсы, настройки |
| **Команда** | Система ввода, очередь действий, макросы |
| **Состояние** | Управление состояниями игры, анимацией персонажа |

**Рисунок 1.2 — Паттерны проектирования в геймдеве**

Стив Макконнелл подчеркивает важность правильного выбора паттернов: «Использование паттернов без понимания их последствий может привести к усложнению кода, а не к его упрощению» [1, с. 156]. Поэтому в GameLib паттерны применяются только там, где они действительно решают конкретную проблему.

### 2.2 Паттерн «Компоновщик» (Composite) в системе компонентов

**Паттерн Компоновщик** (Composite) позволяет сгруппировать объекты в древовидные структуры для представления иерархий «часть-целое». В GameLib этот паттерн реализован в системе компонентов.

**Диаграмма паттерна Компоновщик:**

```
                    ┌─────────────────┐
                    │   <<Component>> │
                    │    Component    │
                    ├─────────────────┤
                    │ +set_game_object│
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────┴────────┐ ┌───┴────┐ ┌───────┴────────┐
     │   LeafComponent │ │Composite│ │  LeafComponent │
     │   AudioSource   │ │GameObject│ │ ImageComponent │
     └─────────────────┘ └────────┘ └────────────────┘
```

**Рисунок 2.2 — Реализация паттерна Компоновщик в GameLib**

**Пример реализации:**

```python
class Component:
    """Базовый компонент - лист в паттерне Компоновщик"""
    def __init__(self):
        self.obj = None
    
    def set_game_object(self, obj):
        self.obj = obj

class GameObject:
    """Игровой объект - контейнер в паттерне Компоновщик"""
    def __init__(self):
        self.components = []  # Список дочерних компонентов
    
    def add_component(self, component):
        """Добавление компонента - метод add из паттерна"""
        if hasattr(component, 'set_game_object'):
            component.set_game_object(self)
        self.components.append(component)
        return self  # Поддержка fluent interface
    
    def update_position(self, new_x, new_y):
        """Обновление позиции с уведомлением всех компонентов"""
        dx, dy = new_x - self.x, new_y - self.y
        self.x, self.y = new_x, new_y
        
        # Компоновщик: операция применяется ко всем дочерним элементам
        for component in self.components:
            if hasattr(component, 'update_position'):
                component.update_position(dx, dy)
```

**Листинг 2.2 — Реализация паттерна Компоновщик**

**Преимущества применения паттерна в GameLib:**

1. **Единообразие** — разработчик работает с объектом и компонентом через единый интерфейс
2. **Гибкость** — компоненты можно динамически добавлять и удалять
3. **Расширяемость** — новые компоненты не требуют изменения существующего кода

Как отмечает Макконнелл: «Компоновщик особенно полезен, когда клиенты не должны различать отдельные объекты и их группы» [1, с. 312]. В GameLib разработчик взаимодействует с GameObject, не заботясь о том, какие компоненты у него есть.

### 2.3 Паттерн «Наблюдатель» (Observer) в системе событий

**Паттерн Наблюдатель** (Observer) определяет механизм подписки, позволяющий одним объектам следить за событиями других объектов. В GameLib этот паттерн реализован в виде глобальной шины событий `EventBus`.

**Структура паттерна Наблюдатель:**

```
┌─────────────┐    subscribe()    ┌─────────────────────────────┐
│             │──────────────────►│                             │
│  Subject    │                   │         Observer            │
│  (Source)   │                   │      (Subscriber)           │
│             │◄──────────────────│                             │
└─────────────┘    notify()       └─────────────────────────────┘
       │
       │ emit()
       ▼
┌─────────────┐
│  EventBus   │
│  (Mediator) │
└─────────────┘
```

**Рисунок 2.3 — Модифицированная реализация Наблюдателя через Mediator**

В классическом паттерне Наблюдатель субъект хранит список наблюдателей. В GameLib используется модифицированная версия с шиной событий, что обеспечивает более слабую связанность.

**Реализация в GameLib:**

```python
class EventBus:
    """Шина событий - реализация паттерна Наблюдатель через Mediator"""
    
    def __init__(self):
        # Ключ: (source, event_type) — может быть None для любого
        self._subscribers = {}
    
    def subscribe(self, source, event_type, callback):
        """Подписка на события"""
        key = (source, event_type)
        if key not in self._subscribers:
            self._subscribers[key] = []
        self._subscribers[key].append(callback)
    
    def emit(self, source, event_type, data=None):
        """Отправка события - уведомление всех наблюдателей"""
        event = Event(source, event_type, data)
        
        # Паттерн цепочки ответственности (Chain of Responsibility)
        patterns = [
            (source, event_type),  # Точное совпадение
            (source, None),        # Все события от источника
            (None, event_type),    # Все события типа
            (None, None)           # Все события
        ]
        
        for pattern in patterns:
            callbacks = self._subscribers.get(pattern)
            if callbacks:
                for cb in callbacks:
                    cb(event)

# Глобальный экземпляр
global_bus = EventBus()
```

### 2.4 Паттерн «Стратегия» (Strategy) в физике и камере

**Паттерн Стратегия** определяет семейство алгоритмов, инкапсулирует каждый из них и делает их взаимозаменяемыми. В GameLib этот паттерн применяется в двух местах: физика может использовать разные стратегии импульса, а камера — разные стратегии слежения.

```python
        self.follow_target = None
        self.follow_strategy = SmoothFollowStrategy()  # Стратегия по умолчанию

        """Изменение стратегии слежения"""
        self.follow_strategy = strategy
    
    def _update_following(self):
        """Обновление слежения - делегирование стратегии"""
        if self.follow_target:
            self.follow_strategy.update(self, self.follow_target)

# Реализация различных стратегий
class SmoothFollowStrategy:
    """Стратегия плавного слежения"""
    def update(self, camera, target):
        target_pos = self._get_target_position(target)
        camera.position += (target_pos - camera.position) * camera.follow_smoothness

class InstantFollowStrategy:
    """Стратегия мгновенного слежения"""
    def update(self, camera, target):
        camera.position = self._get_target_position(target)

class BoundaryFollowStrategy:
    """Стратегия с границами"""
    def update(self, camera, target):
        target_pos = self._get_target_position(target)
        camera.position += (target_pos - camera.position) * 0.05
        
        # Применение границ
        if camera.bounds:
            self._apply_bounds(camera)
```

### 3.1 Алгоритм проверки коллизий AABB

В GameLib используется алгоритм проверки коллизий AABB (Axis-Aligned Bounding Box). Это простейший и наиболее эффективный метод обнаружения столкновений для 2D игр.

**Математическая основа:**

Два прямоугольника A и B пересекаются тогда и только тогда, когда выполняются все четыре условия:

```
A.x < B.x + B.width  И  A.x + A.width > B.x  И
A.y < B.y + B.height И  A.y + A.height > B.y
```

**Визуализация:**

```
        A.x                 A.x + A.width
          │                        │
          ▼                        ▼
    ┌─────┴────────────────────────┴─────┐
    │     ┌─────────────────────┐        │
    │     │        B            │        │
    │     │                     │        │
    │     └─────────────────────┘        │
    └────────────────────────────────────┘
          ▲                        ▲
          │                        │
        B.x                     B.x + B.width
```

**Рисунок 3.1 — Визуализация алгоритма AABB**

**Реализация в GameLib:**

```python
def check_collision(self, obj1, obj2):
    """
    Проверка коллизии между двумя объектами.
    Алгоритмическая сложность: O(1)
    
    Аргументы:
        obj1, obj2: игровые объекты с методом get_coordinates()
    
    Возвращает:
        bool: True если объекты пересекаются
    """
    # Получаем координаты объектов
    coords1 = obj1.get_coordinates()  # [x1, y1, x2, y2]
    coords2 = obj2.get_coordinates()
    
    # Проверка пересечения по оси X и Y
    x_collision = coords1[0] < coords2[2] and coords1[2] > coords2[0]
    y_collision = coords1[1] < coords2[3] and coords1[3] > coords2[1]
    
    return x_collision and y_collision
```

**Листинг 3.1 — Базовый алгоритм AABB**

**Улучшенная версия с дополнительной информацией:**

```python
def get_collision_info(self, obj1, obj2):
    """
    Расширенная проверка коллизии с получением дополнительной информации.
    
    Возвращает:
        dict: {
            'collision': bool,
            'overlap_x': float,  # Величина перекрытия по X
            'overlap_y': float,  # Величина перекрытия по Y
            'side': str          # Сторона столкновения
        }
    """
    c1 = obj1.get_coordinates()
    c2 = obj2.get_coordinates()
    
    # Вычисление перекрытий
    overlap_x = min(c1[2], c2[2]) - max(c1[0], c2[0])
    overlap_y = min(c1[3], c2[3]) - max(c1[1], c2[1])
    
    if overlap_x > 0 and overlap_y > 0:
        # Определение стороны столкновения
        if overlap_x < overlap_y:
            side = "left" if c1[0] < c2[0] else "right"
        else:
            side = "top" if c1[1] < c2[1] else "bottom"
        
        return {
            'collision': True,
            'overlap_x': overlap_x,
            'overlap_y': overlap_y,
            'side': side
        }
    
    return {'collision': False, 'overlap_x': 0, 'overlap_y': 0, 'side': None}
```

**Листинг 3.2 — Расширенный алгоритм AABB**

**Анализ сложности:**
- Временная сложность: O(1) — константное время
- Пространственная сложность: O(1) — не требует дополнительной памяти

Как отмечает Макконнелл: «Простые алгоритмы часто оказываются наиболее эффективными. Сложность должна добавляться только тогда, когда она действительно необходима» [1, с. 523].

### 3.2 Алгоритм физического импульса

Импульс — это мгновенное изменение количества движения объекта. В GameLib реализован алгоритм, который применяет силу к объекту и плавно перемещает его на заданное расстояние.

**Математическая модель:**

```
Импульс: p = m * v
Изменение позиции: Δs = (F * t²) / (2 * m)
```

**Визуализация траектории импульса:**

```
    ┌─────────────────────────────────────────────────────────┐
    │                                                         │
    │    Начало      Шаг 1     Шаг 2     ...      Конец       │
    │      ●─────────●─────────●─────────●─────────●          │
    │      │                                                   │
    │      └───────────────────────────────────────────────────┘
    │                        Время →                          │
    └─────────────────────────────────────────────────────────┘
```

**Рисунок 3.2 — Траектория движения при импульсе**

**Реализация алгоритма:**

```python
def impulse(obj, force, angle_or_vector, mass, on_complete=None):
    """
    Применение импульса к объекту.
    
    Алгоритмическая сложность: O(n) где n = количество шагов (фиксировано = 20)
    
    Алгоритм:
    1. Вычисление направления и расстояния
    2. Разбиение движения на шаги
    3. Пошаговое перемещение объекта
    4. Вызов колбэка при завершении
    """
    if not obj:
        return None
    
    start_pos = obj.get_position()
    is_vector = isinstance(angle_or_vector, Vector2)
    
    # Шаг 1: Вычисление параметров движения
    distance = force * 10 / max(1, mass)
    
    if is_vector:
        vector = angle_or_vector.normalized()
        dx_total = vector.x * distance
        dy_total = vector.y * distance
    else:
        angle_rad = math.radians(angle_or_vector)
        dx_total = distance * math.sin(angle_rad)
        dy_total = distance * math.cos(angle_rad)
    
    # Шаг 2: Параметры анимации
    steps = 20                    # Количество шагов
    speed = max(10, min(50, 100 - force))  # Скорость анимации
    step_dx = dx_total / steps    # Смещение за шаг по X
    step_dy = dy_total / steps    # Смещение за шаг по Y
    current_step = 0
    
    # Шаг 3: Функция пошагового перемещения
    def apply_impulse_step():
        nonlocal current_step
        
        if current_step >= steps:
            # Завершение импульса
            if on_complete:
                on_complete()
            return False, None
        
        # Вычисление новой позиции
        new_x = start_pos[0] + step_dx * (current_step + 1)
        new_y = start_pos[1] + step_dy * (current_step + 1)
        
        # Применение перемещения
        obj.update_position(new_x, new_y)
        current_step += 1
        
        return True, speed
    
    return apply_impulse_step
```

**Листинг 3.3 — Алгоритм импульса**

**График зависимости расстояния от силы и массы:**

```
    │

    │                                 ╱

    │                           ╱

    │                     ╱
 20 ┤                  ╱
    │               ╱
 10 ┤            ╱
    │         ╱
  0 ┼────────╱─────────────────────────────────
    0    2    4    6    8   10   12   14   16
                     Масса
```

- Количество шагов фиксировано (20), поэтому сложность O(1)
- Используется нелинейная зависимость скорости от силы: `speed = max(10, min(50, 100 - force))`
- Бо́льшая сила → меньшее время анимации → более резкое движение

### 3.3 Алгоритм работы 2D камеры со сглаживанием

Камера в GameLib использует алгоритм плавного слежения (smooth following) с возможностью установки границ и эффекта дрожания.

**Математическая модель:**

Плавное слежение реализуется с помощью линейной интерполяции:

```
P_new = P_current + (P_target - P_current) * α
```

где `α` — коэффициент сглаживания (smoothness) в диапазоне (0, 1].

**Визуализация работы камеры:**

```
    Мир (2000x2000)                    Экран (800x600)

│                         │          │             │
│    ┌─────┐              │          │  ┌─────┐    │
│    │Player│             │    →     │  │Player│    │
│    └─────┘              │          │  └─────┘    │
│                         │          │             │
│          Камера ────────┼──────────┼─►           │
│                         │          │             │
└─────────────────────────┘          └─────────────┘
```

**Рисунок 3.4 — Принцип работы камеры**

**Реализация алгоритма:**

```python
        self.follow_target = None
        self.follow_smoothness = 0.1
        self.bounds = None
        self.shake_offset = Vector2(0, 0)
        
        # Кэшированные значения для оптимизации
        self._half_w = width / 2
        self._half_h = height / 2
    
    def update(self, delta_time=1/60):
        """
        Обновление камеры.
        Сложность: O(1) + O(n) где n - количество объектов в сцене
        """
        self._update_following()
        self._update_zoom(delta_time)
        self._update_shake(delta_time)
        self._apply_transform()
    
    def _update_following(self):
        """Алгоритм плавного слежения"""
        if not self.follow_target:
            return
        
        # Получение позиции цели
        target_pos = self._get_target_position()
        target_pos += self.follow_offset
        
        # Линейная интерполяция позиции
        self.position += (target_pos - self.position) * self.follow_smoothness
        
        # Применение границ
        self._apply_bounds()
    
    def _apply_bounds(self):
        """Алгоритм ограничения камеры границами мира"""
        if not self.bounds:
            return
        
        left, top, right, bottom = self.bounds
        half_w = self._half_w / self.zoom
        half_h = self._half_h / self.zoom
        
        # Кламп позиции в допустимые границы
        self.position.x = max(left + half_w, min(right - half_w, self.position.x))
        self.position.y = max(top + half_h, min(bottom - half_h, self.position.y))
    
    def _update_shake(self, delta_time):
        """
        Алгоритм эффекта дрожания камеры.
        Использует генератор случайных чисел для создания эффекта.
        """
        if self.shake_timer <= 0:
            if self.shake_offset.x != 0:
                self.shake_offset = Vector2(0, 0)
            return
        
        self.shake_timer -= delta_time
        
        if self.shake_timer > 0:
            # Линейное затухание интенсивности
            intensity = self.shake_intensity * (self.shake_timer / self.shake_duration)
            # Случайное смещение в пределах [-intensity, intensity]
            self.shake_offset = Vector2(
                (random.random() - 0.5) * intensity * 2,
                (random.random() - 0.5) * intensity * 2
            )
    
    def _apply_transform(self):
        """Применение трансформации камеры ко всем объектам сцены"""
        screen_center = Vector2(self._half_w, self._half_h)
        camera_offset = screen_center - (self.position + self.shake_offset)
        
        for obj in self.scene.gameObj_list:
            if not hasattr(obj, 'rect_id') or not obj.rect_id:
                continue
            
            # Кэширование оригинальных позиций
            if obj not in self._original_positions:
                self._original_positions[obj] = Vector2(obj.x, obj.y)
            
            original = self._original_positions[obj]
            
            # Применение смещения
            self.canvas.coords(
                obj.rect_id,
                original.x + camera_offset.x,
                original.y + camera_offset.y,
                original.x + camera_offset.x + obj.width,
                original.y + camera_offset.y + obj.height
            )
```

### 3.4 Алгоритм диспетчеризации событий

Система событий в GameLib использует алгоритм множественной диспетчеризации с четырьмя паттернами подписки.

**Структура данных:**

```python
_subscribers = {
    (source1, event_type1): [callback1, callback2, ...],
    (source1, None): [callback3, ...],
    (None, event_type2): [callback4, ...],
    (None, None): [callback5, ...]
}
```

**Алгоритм диспетчеризации:**

```
Вход: событие (source, event_type, data)
  │
  ▼
Создать список паттернов:
  patterns = [
    (source, event_type),  # Точное совпадение
    (source, None),        # Все события от источника
    (None, event_type),    # Все события типа
    (None, None)           # Все события
  ]
  │
  ▼
Для каждого pattern в patterns:
  │
  ▼
  Получить callbacks = _subscribers.get(pattern)
  │
  ▼
  Если callbacks существует:
    Для каждого callback в callbacks:
      callback(event)
  │
  ▼
Конец
```

**Рисунок 3.6 — Блок-схема алгоритма диспетчеризации**

**Реализация:**

```python
class EventBus:
    def __init__(self):
        self._subscribers = {}
    
    def subscribe(self, source, event_type, callback):
        """
        Подписка на события.
        Сложность: O(1) амортизированное
        """
        key = (source, event_type)
        if key not in self._subscribers:
            self._subscribers[key] = []
        self._subscribers[key].append(callback)
    
    def unsubscribe(self, source, event_type, callback):
        """
        Отписка от событий.
        Сложность: O(m) где m - количество подписчиков на ключе
        """
        key = (source, event_type)
        if key in self._subscribers:
            subscribers = self._subscribers[key]
            if callback in subscribers:
                subscribers.remove(callback)
    
    def emit(self, source, event_type, data=None):
        """
        Отправка события.
        Сложность: O(k * n) где k = 4 (фиксировано), 
                            n - количество подписчиков
        """
        event = Event(source, event_type, data)
        
        # Паттерны в порядке приоритета
        patterns = [
            (source, event_type),  # Наивысший приоритет
            (source, None),
            (None, event_type),
            (None, None)           # Наинизший приоритет
        ]
        
        for pattern in patterns:
            callbacks = self._subscribers.get(pattern)
            if callbacks:
                # Создаём копию списка для безопасной итерации
                for cb in callbacks[:]:
                    cb(event)
```

**Листинг 3.5 — Алгоритм диспетчеризации событий**

**Пример работы алгоритма:**

```python
# Подписки

bus.subscribe(player, None, on_player_event)       # pattern 2
bus.subscribe(None, "death", on_any_death)         # pattern 3
bus.subscribe(None, None, on_everything)           # pattern 4

# Отправка события

# Вызовутся все 4 колбэка в указанном порядке
```

**Анализ сложности:**
- Подписка: O(1) амортизированное
- Отписка: O(m) где m — количество подписчиков
- Отправка: O(4 * n) = O(n) где n — количество подписчиков

Макконнелл отмечает: «Эффективная структура данных для хранения подписчиков критически важна для производительности событийной системы» [1, с. 478].

### 3.5 Алгоритм управления игровым циклом

Игровой цикл в GameLib реализован в отдельном потоке с возможностью установки целевого FPS.

**Блок-схема игрового цикла:**

```
    ┌─────────────────────────────────────────────────────────┐
    │                      START                              │
    └─────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ running = True  │
                    └─────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
              ┌────►│  while running  │
              │     └─────────────────┘
              │              │
              │              ▼
              │     ┌─────────────────┐
              │     │  game.update()  │
              │     └─────────────────┘
              │              │
              │              ▼
              │     ┌─────────────────┐
              │     │  time.sleep()   │
              │     │  (loop_delay)   │
              │     └─────────────────┘
              │              │
              │              │
              └──────────────┘
```

**Рисунок 3.7 — Блок-схема игрового цикла**

**Реализация:**

```python
class Engine:
    def __init__(self, game=None):
        self.running = False
        self.game_loop_thread = None
        self.game = game
        self.loop_delay = 1.0 / 60.0  # 60 FPS по умолчанию
        
        # Тайминг для стабильного FPS
        self.last_time = 0
        self.frame_count = 0
        self.fps = 0
    
    def start_game_loop(self):
        """Запуск игрового цикла в отдельном потоке"""
        if not self.game or self.running:
            return
        
        self.running = True
        self.game_loop_thread = threading.Thread(target=self._game_loop, daemon=True)
        self.game_loop_thread.start()
    
    def _game_loop(self):
        """
        Основной игровой цикл.
        Использует фиксированную задержку для контроля FPS.
        """
        while self.running and self.game:
            try:
                start = time.time()
                
                # Вызов метода обновления игры
                self.game.update()
                
                # Расчёт времени выполнения
                elapsed = time.time() - start
                
                # Задержка для достижения целевого FPS
                if elapsed < self.loop_delay:
                    time.sleep(self.loop_delay - elapsed)
                
                # Обновление статистики FPS
                self._update_fps_stats()
                
            except Exception as e:
                # Логирование ошибки и завершение цикла
                print(f"Ошибка в игровом цикле: {e}")
                break
        
        self.running = False
    
    def set_fps(self, fps):
        """Установка целевого FPS"""
        if fps > 0:
            self.loop_delay = 1.0 / fps
    
    def _update_fps_stats(self):
        """Обновление статистики FPS для отладки"""
        current = time.time()
        self.frame_count += 1
        
        if current - self.last_time >= 1.0:
            self.fps = self.frame_count
            self.frame_count = 0
            self.last_time = current
```

### А.1 Модуль векторов (vector2.py)

```python
import math

class Vector2():
    def __init__(self, x=0, y=0):
        self.x = x
        self.y = y
    
    @classmethod
    def from_angle(cls, magnitude, angle_degrees):
        angle_rad = math.radians(angle_degrees)
        return cls(magnitude * math.cos(angle_rad), magnitude * math.sin(angle_rad))
    
    def magnitude(self):
        return math.hypot(self.x, self.y)
    
    def normalized(self):
        mag = self.magnitude()
        if mag == 0:
            return Vector2(0, 0)
        return Vector2(self.x / mag, self.y / mag)
    
    def __add__(self, other):
        return Vector2(self.x + other.x, self.y + other.y)
    
    def __sub__(self, other):
        return Vector2(self.x - other.x, self.y - other.y)
    
    def __mul__(self, scalar):
        return Vector2(self.x * scalar, self.y * scalar)
```

```python
        pass
```
